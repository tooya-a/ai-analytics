# スキル作成スキル

このリポジトリ（ai-analytics）の既存スキル（`session-log` / `design-doc` / `create-pr` / `task-report`）と同じ規約で、新しいプロジェクトスキルを雛形生成する。

## テンプレート参照

!`cat .claude/skills/skill-creator-local/references/01-skill-template.md`

## このスキルの設計思想

- **既存スキルと揃える**: フロントマターは `name` / `description` / `tags` の3つ。本文は `## 手順` の番号付きリスト + `## 注意事項` で締めるスタイルに統一する。
- **日本語で書く**: このリポジトリのスキル・CLAUDE.md・README はすべて日本語。出力もデフォルト日本語。
- **`description` がトリガーの本体**: Claude は `description` を見てスキルを起動する。「何をするか」だけでなく「いつ起動すべきか」を含めて、やや押しが強めに書く。
- **小さく始める**: 最初から完璧な手順を網羅しようとしない。3〜5 ステップ + 注意事項で十分。使いながら直す。

## 手順

### 1. ヒアリング

ユーザーに以下を順に確認する（既に会話で判明している項目はスキップしてよい）。

1. **スキル名**（kebab-case の英数字）
   - 例: `code-review` / `release-notes` / `daily-summary`
   - 既存スキル名と重複しないこと。`ls .claude/skills/` で確認する。
2. **このスキルは何をする？**（1〜2 行で）
3. **いつ起動してほしい？**（ユーザーがどんな言葉を言ったとき／どんな文脈のとき）
4. **入力**（会話・git diff・特定のファイル・引数 `$ARGUMENTS` など）
5. **出力**（チャット表示のみ／ファイル保存／外部サービス送信など。保存ならパス）
6. **参照テンプレートが必要か？**（必要なら `references/01-*.md` を別途作る）

### 2. ディレクトリ作成

```bash
mkdir -p .claude/skills/<skill-name>
```

参照テンプレートが必要な場合のみ：

```bash
mkdir -p .claude/skills/<skill-name>/references
```

### 3. SKILL.md を生成

`references/01-skill-template.md` をベースに、ヒアリング結果で埋めた SKILL.md を `.claude/skills/<skill-name>/SKILL.md` に書き出す。

**description の書き方のコツ**:
- 「何をするか」を最初に書く。
- そのあとに「いつ起動すべきか」を具体的な語彙で列挙する（例:「○○について聞かれたら」「○○を作ってと言われたら」）。
- 起動して欲しい場面が複数あるなら、それらを並べる。Claude はこの文を見て起動判断するので、ここがスカスカだと永遠に呼ばれない。

**tags の書き方**: 英語 + 日本語を混ぜてよい（既存スキルがそうしている）。例: `[review, code-quality, コードレビュー]`

### 4. 内容の確認

生成した SKILL.md の中身を表示し、以下を聞く：

> この内容でよいですか？　修正があれば指示してください。**「はい」で確定します。**

- description／手順の妥当性を特に確認する。
- 修正依頼があれば直してから再表示する。

### 5. 完了の報告

- 保存したファイルパスを表示する。
- 次のアクションを案内する：
  - **動作確認**: 新しいセッションを開いて `/<skill-name>` で起動できるか試す（同一セッションでは即時には拾われない場合がある）。
  - **commit**: 「git add して commit しますか？」と聞く。承認されたら：
    ```bash
    git add .claude/skills/<skill-name>/
    git commit -m "feat(skills): add <skill-name> skill"
    ```

## 注意事項

- **既存スキルを上書きしない**: 同名ディレクトリが既にあれば、上書きせず「既に存在します。更新しますか？」と確認する。
- **テンプレート参照は任意**: 短い手順だけのスキルなら `references/` は不要。session-log のようにスキーマや雛形を持ちたいときだけ作る。
- **CLAUDE.md の更新は別作業**: 新しいスキルを作っても、プロジェクト直下の `CLAUDE.md` の「関連スキル」表は自動更新されない。必要なら最後に「CLAUDE.md の関連スキル表も更新しますか？」と提案する。
- **推測で埋めない**: ヒアリングで未確定の項目は `{未確定}` のまま残し、ユーザーに埋めてもらう。とくに「いつ起動するか」が曖昧なまま description を書くと、永久に起動しないスキルが出来上がる。
- **このスキル自身も同じ規約に従う**: 規約を変えたい場合は、既存4スキル＋このスキルをまとめて揃えること。
